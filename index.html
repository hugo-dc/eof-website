Introduction to EVM Object Format

EVM Object Format (EOF) introduces a way to structure EVM bytecode, defining a clear separation of sections (i.e. code, data, etc), it also allow to create new EOF versions without breaking backwards compatibility. Another benefit is that EOF Bytecode is validated at deploy time, instead of validating each time the code is executed as in current EVM bytecode.

EOF EIPs

  • EIP-3541

    Rejects deploying code starting with the byte 0xef. This EIP is already live on Ethereum Mainnet

  • EIP-3540

    EOF - EVM Object Format.
    Introduces the EOF container format with once-off validation at deploy time.

  • EIP-3670

    EOF - Code Validation.
    Defines how EOF bytecode is validated for correctness at deploy time.

  • EIP-4200

    EOF - Static Relative Jumps.
    Introduces new instructions: RJUMP, RJUMPI, and RJUMPV with a signed immediate encoding the jump destination.

  • EIP-4750

    EOF - Functions.
    Introduces EOF Functions as separate code sections. Also introduces two new instructions; CALLF and RETF to call and return from functions.

  • EIP-5450

    EOF - Stack Validation.
    Deploy-time validation of stack usage for EOF Functions.

  • EIP-6206

    EOF - JUMPF intruction.
    Introduces JUMPF instruction for chaining function calls.
